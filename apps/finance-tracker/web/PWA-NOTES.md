# PWA notes (Milestone 15.9)

Lighthouse can't be run reliably headless in this environment, so this milestone
instead ships a **valid, installable PWA** and documents what a human should
verify on a real device / Chrome DevTools. The build emits the manifest, service
worker, and icons (confirmed below); the remaining checks are manual.

## What the build produces

`pnpm --filter finance-tracker-web build` emits into `web/dist/`:

- `manifest.webmanifest` — name, theme/background colours, `display: standalone`,
  and three icons (192, 512, maskable-512).
- `sw.js` + `workbox-*.js` — the generated Workbox service worker.
- `index.html` — with the injected `<link rel="manifest">`, an inline SW
  registration script, `theme-color` (light + dark via `media`), and
  `apple-touch-icon`.
- `icons/` — `icon-192.png`, `icon-512.png`, `icon-maskable-512.png`,
  `apple-touch-icon.png`.

## Service worker caching strategy (15.8)

- **`/api/*` GETs (except auth): `NetworkFirst`** under cache name `api-data`.
  Tries the network first (fresh prices), falls back to the last-cached response
  when offline. `networkTimeoutSeconds: 5`, capped at 64 entries / 24h.
- **`/api/auth/*` (incl. `/api/auth/me`): `NetworkOnly`** — never cached. It is
  ALSO excluded from the NetworkFirst matcher, so a user-identity / token-bearing
  response is never persisted in the SW cache.
- **Cache key normalisation:** a `cacheKeyWillBeUsed` plugin rewrites the key to
  a bare `origin + pathname + search`. The `Authorization` request header is not
  part of the Cache API key anyway (Workbox keys on URL), so tokens cannot leak
  into a cache key.
- **App shell:** precached (`generateSW`) with an `index.html` navigation
  fallback so deep links open offline; `/api/*` is on the navigation denylist.
- The SW is **disabled in `vite dev`** (`devOptions.enabled: false`) so it does
  not shadow the API proxy during local development. Test it against a
  production build: `pnpm --filter finance-tracker-web preview`.

## Manual checks a human should run

Run against a production build served over HTTPS (or `localhost`, which counts as
a secure context). Use `pnpm --filter finance-tracker-web preview`.

1. **Install prompt** — open in Chrome (desktop or Android). DevTools →
   Application → Manifest should show no errors and list all icons. The install
   ("Add to Home screen") affordance should appear; install it and confirm it
   launches standalone with the brand-blue theme.
2. **Offline read** — load the dashboard while online (so prices cache), then go
   offline (DevTools → Network → Offline) and reload. The app shell should open
   and the last-fetched holdings/prices/tiles should render from the `api-data`
   cache. Confirm `/api/auth/me` does NOT serve from cache offline (it should
   fail / show the auth fallback), proving auth was never cached.
3. **Lighthouse (target ≥ 90 PWA / ≥ 90 Performance)** — DevTools → Lighthouse →
   run "Progressive Web App" + "Performance" on the preview build. Expect the
   installability + SW + manifest audits to pass. If Performance dips, the most
   likely culprit is the lazy ECharts chunk (see below) — it should NOT count
   against first load because it is code-split.
4. **Theme-color** — toggle OS light/dark and confirm the address-bar / status
   bar colour follows (`theme-color` is `media`-scoped).

## Performance: ECharts is code-split (confirmed)

The build prints a >500 KB chunk-size warning for `assets/esm-*.js` (~1 MB). That
is the **ECharts** chunk, and it is intentionally **lazy-loaded** (M11): tiles
render charts through `src/tiles/LazyChart.tsx`, which does
`lazy(() => import("echarts-for-react"))`. Verified: the main `index-*.js` bundle
contains **zero** `echarts` references — ECharts only downloads when a chart
first renders. The warning is therefore expected and not a first-load regression.

## Placeholder icons — replace before launch

The icons in `web/public/icons/` are **placeholders** generated by
`scripts/gen-icons.mjs` (a solid brand-blue tile with an ascending-bars glyph).
A designer should replace them with real artwork at the same filenames/sizes
(192, 512, maskable-512, and a 180 apple-touch-icon), then rebuild. Splash
screens (iOS) are intentionally not hand-authored — iOS generates one from the
manifest's `background_color` + icon, which is acceptable for v1.
